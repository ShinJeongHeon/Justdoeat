# PRD — Just do eat 관리자 페이지 (출시 알림 신청 관리)

> 근거 문서: `01-prd-landing.md`(랜딩 목표·범위) · `02-design-landing.md` §11(v3 출시 알림 폼) · `landing/src/components/Waitlist.jsx`(실제 신청 로직) · Supabase `item` 테이블(실제 저장소)
> 산출물: 신청자 데이터를 조회·분석·내보내는 인증된 관리자 웹 페이지

---

## 1. 요약 (Summary)

랜딩페이지의 **출시 알림 신청**으로 Supabase `item` 테이블에 쌓이는 신청자 데이터(이메일·전화번호·직장 동네)를 운영자가 한 화면에서 조회·분석·내보낼 수 있는 **관리자 페이지**를 만든다.

핵심 전략은 "데이터를 쌓는 것에서 **결정을 내리는 것**으로." 랜딩은 방문자에게 `"신청이 많은 동네부터 먼저 엽니다"`라고 약속했다. 관리자 페이지의 존재 이유는 그 약속을 실제로 실행하게 하는 것이다 — **어느 동네에 수요가 몰렸는지 즉시 답하고, 그 동네 신청자에게 출시 알림을 보낼 연락처를 뽑아낸다.**

## 3. 배경 (Background)

- **현황**: 랜딩 v3(`02-design-landing.md` §11)에서 출시 알림 폼을 도입했다. 방문자가 이메일 / 전화번호 / 직장 동네를 남기면 `Waitlist.jsx`가 Supabase `item` 테이블에 `insert` 하고, 재방문 확인용으로 localStorage에도 사본을 남긴다. **데이터는 이미 실제 DB에 쌓이기 시작한다.**
- **문제**: 그런데 그 데이터를 **볼 방법이 없다.** 현재 신청자를 확인하려면 Supabase 대시보드에 직접 로그인해 SQL을 쳐야 한다. 이건 운영자에게 매번 마찰이고, 랜딩이 한 약속("신청 많은 동네부터")을 실행 가능한 정보로 바꾸지 못한다.
- **보안 현황(중요)**: `item` 테이블의 RLS는 현재 **insert 전용**으로 설계됐다(`supabaseClient.js` 주석 참고 — anon 키는 쓰기만). 즉 브라우저에 노출된 anon 키로는 신청자 목록을 **읽을 수 없다.** 이는 의도된 보안 장치다. 관리자 페이지는 이 장치를 우회하지 않고, **인증된 세션에서만 읽기가 열리도록** RLS를 확장해야 한다.
- **가능해진 것**: 저장소가 이미 Supabase이므로, 같은 프로젝트에 Supabase Auth와 read 정책만 추가하면 별도 백엔드 없이 안전한 관리자 페이지를 만들 수 있다.

### 실제 데이터 모델 (source of truth: Supabase `public.item`)

| 컬럼 | 타입 | 폼 필드 | 비고 |
|---|---|---|---|
| `id` | `bigint` (identity) | — | PK |
| `created_at` | `timestamptz` (default `now()`) | — | 신청 시각. 추이·정렬의 기준 |
| `email` | `text` (nullable) | 이메일 | 이메일/전화 중 **최소 하나** 필수(폼 검증) |
| `phone` | `text` (nullable) | 전화번호 | 형식 `^[\d\-+() ]{9,}$` |
| `company` | `text` (nullable) | **직장 동네** | 폼의 `hood` 값. 자유 입력(예: "역삼동", "판교", "을지로") |

> **주의 1 — 컬럼 이름 함정**: 직장 동네가 `company` 컬럼에 저장된다(`Waitlist.jsx`의 매핑). 관리자 UI에서는 반드시 **"직장 동네"**로 표시하고, `company`라는 원본 이름을 노출하지 않는다.
> **주의 2 — 중복/제약 없음**: `item`에는 unique 제약이 없다. 같은 사람이 여러 번 신청하면 중복 행이 생긴다(폼이 재방문 시 localStorage로 막지만, 다른 기기·시크릿 모드는 통과). 지표 계산 시 "고유 연락처" 기준을 고려한다.
> **주의 3 — 유령 테이블**: DB에 `Customer_email` 테이블(unique email)도 있으나 **현재 라이브 폼은 여기에 쓰지 않는다.** 관리자 페이지의 단일 진실은 `item`이다. `Customer_email`은 정리 대상(§8 오픈 이슈)으로 남긴다.

## 4. 목표 (Objective)

**목표**: 운영자가 로그인 한 번으로 "지금 누가, 어느 동네에서, 얼마나 신청했는지"를 즉시 파악하고, 출시 알림을 보낼 연락처를 안전하게 뽑아낼 수 있게 한다.

**왜 중요한가**: 랜딩의 전환(신청)은 그 자체로는 가치가 아니다. 그 신청이 **"어느 동네부터 열지"라는 결정**과 **"그 동네에 알림을 보내는 실행"**으로 이어져야 비로소 가치가 된다. 관리자 페이지는 데이터와 결정 사이의 다리다.

**핵심 결과 (Key Results)** — 프로토타입/1인 운영 단계이므로 정성·즉시측정 위주:

| KR | 목표 | 측정 방법 |
|---|---|---|
| KR1 | 로그인 후 **3클릭·10초 이내**에 "수요 1위 동네"를 답할 수 있다 | 직접 조작 테스트 (v1, 정성) |
| KR2 | 특정 동네/기간 신청자 연락처를 **CSV로 30초 내** 내보낼 수 있다 | 직접 조작 테스트 (v1) |
| KR3 | **인증 없이는 PII(이메일·전화)가 절대 노출되지 않는다** — anon 키·비로그인 접근 시 0건 조회 | RLS 검증 테스트 (v1, 필수 통과) |
| KR4 | 신청 1,000건 규모에서도 목록·검색·집계가 **1초 내** 응답 | 더미 데이터 부하 확인 (v1) |

## 5. 사용자 세그먼트 (User Segments)

관리자 페이지는 외부 사용자가 아니라 **운영 주체**를 위한 도구다. 세그먼트는 "누가 어떤 결정을 위해 이 화면을 여는가"로 정의한다.

- **주 사용자 — 제품 오너 / 파운더(현재 유일)**: 출시 순서·마케팅을 결정한다. 여는 이유 = ① 오늘 얼마나 늘었나(모멘텀 확인) ② 어느 동네가 뜨나(출시 순서) ③ 알림 보낼 명단 뽑기. **모바일에서도 대시보드 수치는 확인**하되, 명단 관리·내보내기는 데스크톱 위주.
- **부 사용자 — (미래) 마케터·운영 조력자**: 여러 명이 될 때 권한 분리가 필요. v1은 **단일 관리자 계정**으로 시작하고, 다계정·권한은 범위 밖(§8).
- **제약**:
  - PII를 다루는 화면이다. **인증은 타협 대상이 아니다.** 비로그인 접근은 데이터가 0건이어야 하며 UI 자체도 차단한다.
  - 별도 서버를 두지 않는다(랜딩과 동일 철학). 인증·권한·읽기는 Supabase Auth + RLS로만 처리한다.
  - 데이터 규모는 당분간 수백~수천 건. 대규모 분석 도구가 아니라 **가볍고 빠른 운영 콘솔**이면 충분하다.

## 6. 가치 제안 (Value Propositions) — 관리자가 해결할 일(Jobs)

| 일(Job) | 얻는 것 | 없으면 겪는 고통 |
|---|---|---|
| ① **모멘텀 파악** — "신청이 늘고 있나?" | 총 신청 수 + 오늘/이번 주 신규 + 일별 추이 | Supabase에 매번 SQL. 성장 감각을 잃음 |
| ② **출시 순서 결정** — "어느 동네부터 열까?" | **동네별 신청 수 랭킹**(정규화 집계) | 랜딩의 약속을 실행할 근거가 없음 |
| ③ **알림 실행** — "그 동네에 어떻게 연락하지?" | 필터된 연락처 **CSV 내보내기**(이메일/전화) | 손으로 복사, 누락·오타 |
| ④ **데이터 위생** — "중복·스팸을 치우자" | 검색·정렬·행 삭제, 중복 표시 | 오염된 명단으로 잘못된 결정 |
| ⑤ **안전 보관** — "PII가 새지 않게" | 인증 게이트 + RLS로 읽기 제한 | 신청자 개인정보 유출 리스크 |

**핵심 통찰**: 이 다섯 중 무게중심은 **②동네 랭킹**이다. 랜딩이 방문자에게 한 유일한 구체적 약속("신청 많은 동네부터")의 실행 도구이기 때문이다. 대시보드의 1순위 위젯은 동네 랭킹이어야 한다.

### 7.2 핵심 기능

1. **인증 게이트 (필수·최우선)**
   - Supabase Auth로 로그인. 세션 없으면 로그인 화면만, 모든 데이터 호출 차단.
   - **RLS 확장**: `item` 테이블에 `authenticated` 롤 대상 `SELECT`(및 `DELETE`) 정책 추가. anon 롤은 기존대로 `INSERT`만. → 브라우저 anon 키로는 여전히 목록 조회 불가(KR3).
   - 클라이언트에 service_role 키를 **절대 넣지 않는다.** 읽기 권한은 로그인 세션의 JWT + RLS로만.

2. **대시보드 지표**
   - 지표 카드: 총 신청 수 / 오늘 신규 / 이번 주(7일) 신규 / 연락수단 분포(이메일 보유 %, 전화 보유 %).
   - **동네별 신청 랭킹**(1순위 위젯): `company` 기준 집계, 가로 막대. 상위 N개 + "기타" 접기.
   - 일별 신청 추이: 최근 30일 `created_at` 일 단위 집계 라인.
   - 집계는 가능하면 Postgres 뷰/RPC(`group by`)로 서버에서 처리 → 클라이언트로 전 행을 내려받지 않음(KR4).

3. **동네 정규화 (집계 정확도의 핵심)**
   - 자유 입력이라 "역삼동 / 역삼 / 역삼동 " 이 제각각 집계될 위험.
   - v1: 표시·집계 시 `trim` + 내부 공백 정리 + 대소문자 정규화 수준의 **가벼운 정규화**. 원본은 보존하고 정규화 키로만 그룹핑.
   - v2: 동네 별칭 매핑 테이블(수동 병합 UI). v1에선 병합이 필요한 상위 동네만 눈으로 확인.

4. **신청자 목록**
   - 컬럼: 신청일시 · 이메일 · 전화 · 직장 동네. `created_at` 최신순 기본 정렬, 헤더 클릭 정렬.
   - **검색**: 이메일/전화/동네 부분일치(대소문자 무시).
   - **필터**: 동네(집계에서 선택), 기간(오늘/7일/30일/사용자 지정), 연락수단(이메일만/전화만/둘 다).
   - **페이지네이션**: range 기반(`.range()`), 페이지당 50건. 1,000건+에서도 가벼움.
   - **행 관리**: 다중 선택 → **소프트 삭제**(`deleted_at` 표시, 물리 삭제 아님 — §8 확정). 삭제 전 인페이지 확인 모달(**브라우저 confirm() 금지**). 삭제된 항목은 기본 목록에서 제외.
   - **중복 표시**: 같은 이메일/전화가 2건 이상이면 뱃지로 표시(정리 판단 보조).

5. **CSV 내보내기**
   - **현재 필터·검색이 적용된 결과 전체**를 CSV로. 컬럼: 신청일시, 이메일, 전화, 직장 동네.
   - 인코딩 UTF-8 BOM(엑셀 한글 깨짐 방지). 파일명에 날짜·필터 요약.
   - 이것이 "출시 알림을 실제로 보내는" v1의 실행 경로다(발송 자체는 외부 도구, §8 참고).

### 7.3 디자인 방향 — "도구는 도구답게"

랜딩은 **감성·설득**이지만 관리자는 **도구**다. 목적이 다르므로 얼굴도 다르게 간다.

| 랜딩(감성) | 관리자(도구) |
|---|---|
| 여백·모션·펀치라인으로 마음을 움직임 | **데이터 밀도·스캔 속도·명료함** 우선 |
| Black Han Sans 대형 헤드라인 | 헤드라인 최소화, 숫자·표 중심(`tabular-nums`) |
| "결정의 순간" 3D 시퀀스 | 상시 애니메이션 없음, 상태 전환 피드백만 |

단, **팔레트·타이포 토큰은 v2 시스템을 재사용**해 같은 제품의 도구임이 드러나게 한다(배경 `#F2F3EF`, 카드 `#FFF`, 잉크 `#1D2A38`, 브랜드 레드 `#E0431F`, 긍정 `#1E7A4C`, 정보 네이비 `#31567F`). 강조는 색으로, 장식 이모지 금지.

#### 자기검증 — 피한 관리자 대시보드 디폴트

| 흔한 어드민 디폴트 | 이 페이지의 선택 |
|---|---|
| 사이드바 + 6개 메뉴 + 빈 위젯 12개 | 단일 페이지 2섹션. 실제 있는 데이터(신청자)만 다룸 |
| 모든 지표를 동등하게 나열 | **동네 랭킹을 1순위**로 — 랜딩 약속과 직결 |
| 화려한 도넛·게이지 차트 | 가로 막대 랭킹 + 단순 라인 하나. 읽기 속도 우선 |
| 원본 컬럼명(`company`) 그대로 노출 | 사용자 언어("직장 동네")로 표시 |
| 클라이언트에서 전 행 로드 후 JS 집계 | 서버(뷰/RPC) 집계 + range 페이지네이션 |

### 7.4 기술

- **스택**: 랜딩과 동일 — Vite + React + `@supabase/supabase-js`. `landing/`과 분리된 **`admin/`** 앱(또는 보호된 라우트). 배포도 별도 경로.
- **인증**: Supabase Auth(이메일+비밀번호 권장, 단일 관리자 계정). 세션은 supabase-js가 관리.
- **데이터 접근**: 로그인 세션 JWT로 `item` 조회. service_role 키는 클라이언트에 두지 않음.
- **DB 변경(마이그레이션)**:
  1. `item`에 `authenticated` 롤 `SELECT`·`UPDATE` 정책 추가(anon은 기존 insert 정책 유지). UPDATE는 소프트 삭제(`deleted_at` 갱신)용.
  2. `item`에 `deleted_at timestamptz null` 컬럼 추가(소프트 삭제, §8 확정). 기본 조회는 `deleted_at is null` 필터.
  3. 집계용 뷰 또는 RPC: 동네별 카운트, 일별 카운트, 요약 지표(모두 `deleted_at is null` 기준).
  4. 조회 성능을 위한 인덱스: `created_at`, `company`(정규화 표현식 인덱스 검토).
- **환경변수**: 랜딩과 같은 `VITE_SUPABASE_URL` / `VITE_SUPABASE_PUBLISHABLE_KEY` 재사용. 관리자 전용 비밀키는 도입하지 않음.

### 7.5 접근성·품질

- 표는 시맨틱 `<table>`, 정렬 헤더 `aria-sort`, 로딩·빈 상태·에러 상태 명시.
- 지표·랭킹 변경은 조용히, 삭제 등 파괴적 동작만 인페이지 확인.
- **브라우저 네이티브 다이얼로그(alert/confirm/prompt) 사용 금지** — 인페이지 모달로 대체.
- 색만으로 정보 전달 금지(랭킹 막대에 수치 병기).
- 반응형: 데스크톱 우선(운영 작업), 모바일은 대시보드 지표까지 확인 가능하게.

### 7.6 가정 (검증 필요)

- [ ] 단일 관리자 계정으로 충분하다(당분간 협업자 없음) → 사용자 확인

## 8. 릴리스 (Release)

**v1 (첫 버전 — 핵심 운영 콘솔)**
- 로그인(인증 게이트) + `item` 읽기 RLS 확장 + anon 읽기 차단 검증(KR3)
- 대시보드: 총계·신규·연락수단 분포 + **동네별 랭킹** + 일별 추이
- 목록: 검색·필터·정렬·페이지네이션 + 행 삭제(확인 모달) + 중복 표시
- CSV 내보내기(필터 반영, UTF-8 BOM)
- 완료 기준: KR1~KR4 통과, 더미 1,000건 부하 확인, 비로그인 0건 조회 확인

**v2 (후속)**
- 동네 별칭 병합 UI(정규화 매핑 테이블)
- 휴지통 — 소프트 삭제 항목 복구/영구삭제 UI
- 발송 상태 관리("알림 보냄" 표시), 세그먼트 저장
- 다계정·권한(마케터 롤), 감사 로그
- (검토) 이메일/SMS 발송 연동 — 외부 서비스(예: Resend/솔라피) 도입 여부 판단 후

**범위 제외 (v1에서 하지 않는 것)**
- 실제 알림 발송(메일/SMS) 자동화 — v1은 내보내기까지만
- 다계정·세밀한 권한 관리
- 랜딩 콘텐츠·A/B 편집 등 CMS 기능
- `item` 외 테이블 관리(단, 아래 오픈 이슈의 `Customer_email` 정리 제외)

### 확정 결정 (사용자 위임 → 이상적 기준으로 확정, 2026-07-08)

> 판단 기준: **① 작동 중인 랜딩 폼·RLS를 건드리지 않는다, ② 진짜 신청자를 실수로 잃는 비가역적 손실을 막는다, ③ 1인 운영에 맞는 단순함.**

1. **저장소·컬럼명 — 현행 유지**: 관리자·랜딩 모두 `item` 테이블을 단일 진실로 둔다. 컬럼명 `company`도 그대로 두고 UI에서만 "직장 동네"로 표기한다. → 이미 배포되어 데이터를 받는 폼(`Waitlist.jsx`)과 그 RLS를 재작성하는 리스크를 감수할 이유가 없다. (테이블/컬럼 리네이밍은 언젠가 할 수 있는 정리 작업이지만 v1의 일이 아니다.)
2. **`Customer_email` 테이블 — 손대지 않음**: 라이브 폼이 쓰지 않는 미사용 테이블(현재 0행). 삭제하지 않고 그대로 둔다 — 내가 만들지 않았고, 지우는 것은 비가역적이며, 관리자는 `item`만 읽으면 되므로 존재해도 무해하다. (문서상 deprecated로만 표시.)
3. **인증 — 이메일+비밀번호, 단일 관리자 계정**: 계정은 Supabase 콘솔에서 1회 수동 생성. 매직링크는 메일 발송(도메인·SMTP) 설정 부담이 있어 1인 운영에는 과하다.
4. **삭제 — 소프트 삭제**: `deleted_at` 컬럼으로 숨김 처리하고 물리 삭제는 하지 않는다. 실수로 지운 신청자는 다시 신청하지 않으므로 복구 불가 손실이 가장 비싼 오류다. 개인정보 완전 삭제(잊힐 권리) 요청이 오면 그때 수동 물리 삭제로 대응.
5. **발송 범위 — v1은 CSV까지**: 메일/SMS 자동 발송은 제공자 연동·수신거부·템플릿 부담이 커 v2로 미룬다. v1의 실행 경로는 "필터 → CSV 내보내기 → 외부 도구로 1회 발송".
