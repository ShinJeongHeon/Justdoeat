## 1. 디자인 컨셉

**"데이터가 아니라 결정을 보여주는 콘솔."**

랜딩은 방문자의 *마음*을 움직이는 페이지였다. 관리자는 운영자가 *결정*을 내리는 도구다.
그래서 얼굴은 같은 제품이되(토큰·로고·어휘 계승), 목적이 다르므로 태도가 다르다 —
여백·모션·펀치라인 대신 **데이터 밀도 · 스캔 속도 · 명료함**이 지배한다.

이 콘솔이 대답해야 하는 단 하나의 질문은 PRD가 못 박았다: **"어느 동네부터 열까?"**
랜딩이 방문자에게 한 유일한 구체적 약속 — `"신청이 많은 동네부터 먼저 엽니다"` — 의
실행 도구이기 때문이다. 그래서 이 페이지의 무게중심은 지표 나열이 아니라 **동네 랭킹**이고,
랭킹은 중립적 막대차트가 아니라 **"먼저 열 동네"의 순번**으로 프레이밍된다.

- 정적인 부분은 제품(v2 시스템)과 같은 얼굴로, 그러나 더 조밀하게.
- 이 페이지의 서사는 하나뿐 — **랭킹(무엇을) → 명단(누구에게) → 내보내기(어떻게)** 로 내려가는 다리.
- 장식은 없다. 강조는 색으로, 리듬은 숫자로. 랜딩의 3D·모션 시퀀스는 여기 없다.

### 랜딩과 관리자 — 같은 제품, 다른 태도 (PRD §7.3 확장)

| | 랜딩 (감성·설득) | 관리자 (도구·결정) |
|---|---|---|
| 목적 | 마음을 사로잡아 신청하게 | 신청을 읽어 **출시 순서·명단**으로 |
| 헤드라인 | Black Han Sans 대형 | 최소화 — 숫자·표 중심(`tabular-nums`) |
| Black Han Sans | 히어로·섹션 헤드 전반 | **결정을 이름 짓는 순간에만** (워드마크 + 1위 동네명) |
| 잉크(`--ink`) | 마지막 밴드 1회 반전 | **상시 상단바 크롬** — "결정이 내려지는 곳" |
| 모션 | "결정의 순간" 3D 시퀀스 | 상시 모션 없음. 상태 전환 피드백만 |
| 밀도 | 넉넉한 여백(읽는 페이지) | 조밀(스캔하는 도구) |

## 2. 디자인 원칙 (v2 계승 + 도구 확장)

1. **얼굴 일치, 태도 전환**: 팔레트·로고·서체·radius·포커스 링은 랜딩과 100% 동일. 달라지는 것은 밀도와 위계뿐. 운영자가 "같은 제품의 뒷문"에 들어왔다고 느끼게 한다.
2. **어휘는 사용자 언어**: 원본 컬럼명(`company`)을 절대 노출하지 않는다 — 항상 **"직장 동네"**. 삭제는 **"숨기기"**(복구 가능함을 함의), 내보내기 버튼과 완료 토스트는 같은 단어를 쓴다. 시스템 용어가 아니라 운영자가 통제하는 대상의 이름으로 라벨링.
3. **강조는 아껴서 한 곳에**: 브랜드 레드(`--brand`)는 "결정의 색"이다 — 1위(먼저 열 동네) 막대와 활성 필터 칩에만. 모든 막대를 빨갛게 칠하면 빨강이 의미를 잃는다.
4. **색만으로 정보 전달 금지**: 모든 막대·상태에 수치·라벨 병기(PRD 7.5). 순위는 색이 아니라 **길이 + 숫자 + 이름**으로 전달.
5. **모션은 피드백일 때만**: 상시 루프·페이지 로드 안무·패럴랙스 금지. 허용 = 호버/프레스 마이크로 피드백, 정렬 방향 전환, 모달 열림, 토스트, 그리고 **동네를 "실행 대상으로 고를 때"의 스푼체크 체크 드로잉**(랜딩 "결정 완료" 도장과 같은 어휘).
6. **브라우저 네이티브 다이얼로그 금지**: `alert/confirm/prompt` 사용하지 않는다 — 파괴적 동작은 인페이지 모달로만(PRD 7.5).

## 3. 토큰 시스템

### 3.1 색 — v2 그대로, 신규 색 없음 (관리자 적용 규칙만 신설)

토큰 값은 `landing/src/styles.css`와 동일. 아래는 **이 콘솔에서의 역할** 정의다.

| 토큰 | 값 | 관리자에서의 역할 |
|---|---|---|
| `--bg` | `#F2F3EF` | 페이지 캔버스 |
| `--surface` | `#FFFFFF` | 카드·표·패널 |
| `--ink` | `#1D2A38` | **상단바 크롬(반전)** · 본문 텍스트 · 모달 오버레이 딤 |
| `--sub` | `#5D6975` | 보조 텍스트·라벨·숨김 처리된 행 |
| `--line` / `--line2` | `#E4E6E0` / `#CFD3CA` | 카드·표 보더, 행 구분선 |
| `--brand` / `--brand-deep` | `#E0431F` / `#B5300F` | **1위(먼저 열 동네) 막대 · 활성 필터 칩 · 주 액션 강조.** "결정의 색" — 아껴 쓴다 |
| `--brand-soft` | `#FCEBE4` | 활성 필터/선택 행 배경 |
| `--ok` / `--ok-soft` | `#1E7A4C` / `#E3F1E8` | 증가 델타(`+37`), 이메일 보유 등 긍정 지표 |
| `--warn` / `--warn-soft` | `#AF7712` / `#F6EDD7` | **중복 뱃지**, 주의 데이터 |
| `--navy` / `--navy-soft` | `#31567F` / `#E9EEF5` | 추이 라인 · 정렬 활성 표시 · 포커스 링 · 정보 칩 |

**관리자 전용 확장(색이 아니라 사용법)**:
- **상단바 반전**: `--ink`를 배경으로 상시 사용(랜딩은 마지막 밴드에서만 반전). 위에 스푼체크·워드마크·유틸 버튼은 흰색/ghost.
- **랭킹 2위 이하 막대**: 브랜드 레드가 아니라 `--ink` 저투명 채움(`rgba(29,42,56,.14)`) 또는 `--navy-soft` 계열의 **차분한 단색**. 순위 강조는 1위에만 레드.
- **숨김(소프트 삭제) 행**: 취소선 없이 `--sub` 텍스트 + 40% 불투명 — "제거"가 아니라 "치워둠"으로 읽히게.

### 3.2 타입 스케일 — 랜딩보다 한 단계 작고 조밀 (읽는 페이지가 아니라 스캔하는 도구)

랜딩은 앱보다 크게 갔다면, 관리자는 **데이터 밀도** 쪽으로 되돌린다.
Black Han Sans는 **결정을 이름 짓는 두 순간에만** 등장한다 — 아이덴티티 워드마크와 **1위 동네명**.
나머지 모든 수치는 랜딩 `.demo-score` 관례(`Pretendard 800 + tabular-nums`)를 그대로 따른다.

| 역할 | 서체 | 크기 | 비고 |
|---|---|---|---|
| 아이덴티티 워드마크 (상단바·로그인) | Black Han Sans | 19–22px | 랜딩 `.wordmark` 재사용 |
| **1위 동네명** (먼저 열 동네) | Black Han Sans | `clamp(22px, 3.2vw, 30px)` | 랜딩 `.demo-name`과 같은 역할 — "결정된 대상"을 이름 짓는다 |
| KPI 수치 | Pretendard 800 | `clamp(24px, 3.2vw, 34px)` | `tabular-nums` |
| 델타 (`+37`, `82%`) | Pretendard 700 | 12–13px | 부호를 색으로: 증가 `--ok` |
| 랭킹 동네명 (2위~) | Pretendard 800 | 15px | |
| 랭킹·표 수치 | Pretendard 700/800 | 13–14px | `tabular-nums` |
| 표 헤더 | Pretendard 800 | 12px | `letter-spacing .04em` |
| 표 본문 | Pretendard 400/600 | 13.5px | 이메일·전화는 `font-variant-numeric: tabular-nums` 권장 |
| 아이브로우·그룹 라벨 | Pretendard 800 | 11.5px | `letter-spacing .12em`, `--sub` |

Black Han Sans는 단일 웨이트 → 강조는 굵기가 아니라 **색**(브랜드 레드)으로.

### 3.3 간격·형태 — 도구답게 조밀하게

- 페이지 콘텐츠 최대폭: **1200px**(표가 숨 쉴 폭). 좌우 패딩 20px.
- 카드 간격: 16px(랜딩 섹션 리듬 72/112px보다 훨씬 조밀 — 한 화면에 지표+랭킹이 같이 보이게).
- 카드 내부 패딩: 18–20px. 표 행 높이: **44px**(스캔 밀도 ↔ 터치 타깃 균형), `tabular-nums` 정렬.
- radius: 카드 16px, 버튼·인풋 10–12px, 칩·뱃지 999px (v2 동일). 범위 밖 라운딩 금지.
- 그림자: `--shadow` v2 값 그대로. 랭킹 히어로 패널만 은은하게 한 단계, 모달은 `0 16px 48px rgba(0,0,0,.28)`.

### 3.4 모션 토큰 — 최소만 재사용

| 토큰 | 값 | 관리자 용도 |
|---|---|---|
| `--dur-micro` | 150ms | 호버·프레스·칩 토글·정렬 전환·행 하이라이트 |
| `--dur-reveal` | 450ms | 로딩→데이터 페이드, 모달·토스트 등장, 체크 드로잉 |
| `--ease-out` | `cubic-bezier(.2,.7,.3,1)` | 기본(전부 이걸로) |

랜딩의 `--dur-flip`·`--ease-flip`·3D 관련 토큰은 **쓰지 않는다**(관리자에 3D·플립 없음).

## 4. 레이아웃

### 4.0 로그인 게이트 (비인증 시 이 화면만)

PII를 다루는 화면의 문. 화려하지 않게, 그러나 "운영자 전용"이라는 무게가 느껴지게.

```
┌───────────────────────────────────────┐
│                                       │
│              (●) 스푼체크 배지         │  ← SpoonBadge(고추장 레드 원)
│            JUST DO EAT · 관리자         │  ← 워드마크 Black Han Sans
│                                       │
│     ┌───────────────────────────┐     │
│     │ 이메일                     │     │  ← .field 스타일 재사용
│     │ [                       ]  │     │
│     │ 비밀번호                   │     │
│     │ [                       ]  │     │
│     │ [ ✓  로그인 ]  ← cta full  │     │
│     └───────────────────────────┘     │
│   운영자 전용 — 신청자 개인정보를      │  ← 보안을 기능으로 명시
│   다루는 화면입니다.                   │
└───────────────────────────────────────┘
```

- 배경 `--bg`, 중앙 화이트 카드(min 360px). 랜딩 폼(`.field`, `.cta.full`) 컴포넌트 그대로.
- 로그인 실패: `이메일 또는 비밀번호가 맞지 않아요.` (어느 쪽이 틀렸는지 밝히지 않음 — 보안).
- 세션은 supabase-js가 관리. 성공 시 콘솔로 전환, 세션 만료 시 이 화면으로 되돌아옴.

### 4.2 모바일 (<1020px) — 대시보드 우선, 목록은 읽기 카드

```
┌────────────────────┐
│ (✓) 관리자   [⋯]   │  ← 상단바(유틸은 메뉴로 접기)
│ ┌ 총신청 ┐┌ 오늘 ┐ │  ← 스트립 2×2
│ ┌ 이번주 ┐┌ 수단 ┐ │
│ ┌ 먼저 열 동네 ───┐ │  ← 랭킹(막대 세로 스택)
│ │ 역삼동   ███ 312│ │
│ │ 판교     ██  201│ │
│ └─────────────────┘ │
│ ┌ 최근 30일 추이 ─┐ │
│ └─────────────────┘ │
│ ─ 신청자 ─────────  │  ← 목록 = 스택 카드(읽기)
│ ┌ 07-08 · 역삼동  ┐ │
│ │ a@b.com / 010-… │ │
│ └─────────────────┘ │
│ [내보내기]          │  ← 관리(숨기기)는 데스크톱 위주
└────────────────────┘
```

- PRD §5: **모바일은 대시보드 지표 확인까지** 우선. 명단 관리·다중 삭제는 데스크톱 위주.
- 표는 모바일에서 가로 스크롤 대신 **스택 카드**로 붕괴(신청일시·연락처·동네·중복 뱃지). 검색/필터는 접이식 바.
- DOM 순서 = 모바일 읽기 순서(랜딩과 동일 원칙): 스트립 → 랭킹 → 추이 → 목록.

## 5. 화면별 스펙

### [1] 상단바 — 잉크 콘솔 크롬

- 배경 `--ink`, `position: sticky; top: 0`, 높이 ~56px. 랜딩 헤더보다 낮게(도구는 크롬을 아낀다).
- 왼쪽: `SpoonMark`(흰색, `currentColor`) + 워드마크 `JUST DO EAT` + 얇은 구분점 + `관리자` 라벨(`--sub`보다 밝은 흰 70%).
- 오른쪽: `[내보내기]`(ghost 버튼, 랜딩 `.linkbtn.ghost` 재사용) · `[로그아웃]`(ghost). 파괴적이지 않은 두 액션만.
- 유일한 반전 구간을 상시로 승격한 이유: 랜딩에서 잉크는 "고민은 여기까지 — 결정" 밴드였다. 관리자는 그 결정이 *실제로* 내려지는 곳이므로 그 색을 크롬으로 입는다. 장식이 아니라 의미의 연장.

### [2] 모멘텀 스트립 — 지표 4개 (강등된 KPI)

- 한 줄 4열(모바일 2×2). 카드가 아니라 **얇은 타일** — 배경 `--surface`, 보더 `--line`, 패딩 14px 16px. 화면 상단 절반을 먹지 않는다(자기검증: "동일 크기 KPI 카드 4개" 회피).
- 각 타일: 라벨(아이브로우, `--sub`) + 수치(Pretendard 800, `tabular-nums`) + 델타/보조.
  1. **총 신청** — 고유 연락처 기준 병기 옵션(`1,240 · 고유 1,198`). 중복 존재를 숨기지 않음(PRD 주의 2).
  2. **오늘 신규** — `+37` (`--ok`, 자정 기준).
  3. **이번 주** — `+182` (최근 7일).
  4. **연락수단** — `메일 82% · 전화 41%`(두 줄). 발송 채널 판단용.
- 수치 갱신은 조용히(모션 없음). 로딩 중엔 `—`.

### [3] 먼저 열 동네 — 동네 랭킹 (★ 시그니처 히어로 위젯)

이 페이지의 존재 이유. 중립적 "동네별 신청 수"가 아니라 **"먼저 열 동네의 순번"**으로 프레이밍한다 —
순서가 곧 출시 우선순위이고, 그것이 랜딩이 방문자에게 한 약속의 실행이기 때문이다.
(랭킹/번호는 *진짜 순서가 정보를 담을 때만* 정당하다 — 여기선 담는다.)

- 패널: 화이트, radius 16, `--shadow` 한 단계. 대시보드에서 가장 넓은 자리(2fr).
- 아이브로우: `먼저 열 동네` · 보조 한 줄 `신청 많은 순서대로 — 랜딩에서 약속한 그대로.`
- **1위 행(먼저 엽니다)**: 확대. 작은 태그 `1순위`(브랜드 레드 필) + **동네명 Black Han Sans**(`clamp(22,3.2vw,30px)`) + 가로 막대(**`--brand`**) + 수치 `312명`(Pretendard 800, `tabular-nums`) + 액션 `이 동네 명단 보기 →`.
- **2위 이하 행**: 조밀. 순위 pill(`--bg`/`--sub`) + 동네명(Pretendard 800 15px) + 막대(**차분한 `--ink` 저투명**) + 수치 + hover 시 `명단 보기 →`.
- **막대 규칙**: 폭 = `count / max`. 모든 막대에 수치 병기(색만으로 전달 금지). 1위만 레드, 나머지는 길이·숫자로 순위 전달.
- **롱테일 접기**: 상위 N개(기본 6) 노출 + `기타 12개 동네 펼치기 ▾`. 도넛/게이지 없음(자기검증).
- **정규화 투명성**(PRD 3·주의 2): 표기 흔들림("역삼동/역삼/역삼동 ")을 정규화 키로 합쳐 집계하되, 병합된 동네엔 작은 `·병합 3` 힌트 → hover/tap 시 원본 표기 목록을 툴팁으로. 원본은 보존, 집계만 정규화.
- **집계 출처**: Postgres 뷰/RPC(`group by`, `deleted_at is null`). 전 행을 내려받아 JS로 집계하지 않는다(KR4·자기검증).

**상호작용(시그니처의 핵심 — §6):** 행 클릭 → 아래 목록이 그 동네로 필터링되고 페이지가 목록으로 스크롤. 클릭 순간 1위 태그/선택 행에 **스푼체크 체크 드로잉**(랜딩 "결정 완료"와 같은 어휘) — "이 동네를 실행 대상으로 골랐다".

### [4] 최근 30일 추이 — 슬림 라인 하나

- 랭킹 오른쪽(1fr). 작은 카드. 아이브로우 `최근 30일`.
- **단일 라인**(`--navy`), 축·격자 최소(베이스라인만). 오늘 지점만 점 강조 + `+37` 라벨.
- 화려한 차트 라이브러리 금지 — 인라인 SVG `path` 하나. 읽기 속도 우선(자기검증).
- 데이터 `created_at` 일 단위 집계(뷰/RPC, `deleted_at is null`).

### [5] 신청자 목록 — 표

- **컬럼**: 신청일시 · 이메일 · 전화 · **직장 동네** · (행 메뉴). `created_at` 최신순 기본.
- **정렬**: 헤더는 `<button>`, 클릭 시 방향 토글, `aria-sort` 갱신, 활성 헤더에 `--navy` 캐럿. 신청일시·직장 동네 정렬 지원.
- **검색**: 상단 단일 입력 — 이메일/전화/동네 부분일치(대소문자·공백 무시). 입력 디바운스, 결과 수 `aria-live="polite"`.
- **필터**: `동네▾`(랭킹에서 선택 가능) · `기간▾`(오늘/7일/30일/사용자 지정) · `연락수단▾`(이메일만/전화만/둘 다). 활성 필터는 브랜드-소프트 칩으로 표시 + `✕`로 개별 해제. 랭킹에서 내려온 동네 필터는 `선택됨: 역삼동 ✕`로 목록 헤더에 명시.
- **페이지네이션**: `.range()` 기반, 50건/페이지. 1,000건+에서도 가벼움(KR4). 하단 `‹ 1 2 3 ›`.
- **중복 표시**(PRD 주의 2): 같은 이메일/전화가 2건 이상이면 해당 셀에 `중복` 뱃지(`--warn-soft`/`--warn`). 정리 판단 보조 — 자동 삭제는 하지 않음.
- **행 관리**: 좌측 체크박스 다중 선택 → `선택 숨기기`. 행 끝 `⋯` 메뉴에 단건 숨기기.
- **숨김 행 표기**: 소프트 삭제된 항목은 기본 목록에서 제외(`deleted_at is null`). 방금 숨긴 건은 되돌리기 토스트로 즉시 복구 가능(§5-[6]).

### [6] 숨기기 확인 — 인페이지 모달 (브라우저 confirm 금지)

- 트리거: `선택 숨기기` 또는 행 `⋯ › 숨기기`.
- 오버레이: `--ink` 50% 딤 + 중앙 화이트 카드(radius 16, `0 16px 48px`). `role="dialog"` `aria-modal="true"`, 포커스 트랩, `Esc` 닫기, 닫으면 트리거로 포커스 복귀.
- 카피(소프트 삭제의 성격을 정확히):
  - 제목: `이 신청자를 숨길까요?` / 다중: `선택한 32명을 숨길까요?`
  - 본문: `목록에서 감춰집니다. 데이터는 삭제되지 않고, 방금 숨긴 항목은 되돌릴 수 있어요.`
  - 버튼: `취소`(ghost) · `숨기기`(`--ink` 채움 — 파괴적이지만 가역적이라 레드까진 아님).
- 실행 후: 토스트 `32명을 숨겼어요.` + `되돌리기`(세션 내 즉시 복구). 완전 삭제(잊힐 권리)는 수동 대응(PRD 확정 4) — v1 UI엔 없음, 휴지통은 v2.

### [7] CSV 내보내기

- 진입: 상단바 `[내보내기]` + 목록 하단 `[필터 결과 CSV 내보내기]` (둘 다 같은 동작·같은 단어).
- **현재 필터·검색이 적용된 결과 전체**를 내보낸다 — 화면의 50건이 아니라 조건에 맞는 전 행을 재조회(페이지네이션 제외). 이것이 "그 동네에 알림 보내기"의 v1 실행 경로.
- 컬럼(사용자 언어): `신청일시, 이메일, 전화, 직장 동네`. 원본 `company` 노출 금지.
- 인코딩 **UTF-8 BOM**(엑셀 한글 깨짐 방지). 파일명에 날짜·필터 요약: `justdoeat_신청자_역삼동_2026-07-08.csv`.
- 성공 토스트: **스푼체크 체크 드로잉** + `역삼동 312명을 내보냈어요.` (내보내기 버튼과 같은 단어로 닫음).

### 상태 디자인 (로딩·빈·에러 — PRD 7.5)

| 상태 | 화면 | 카피(운영자 언어, 실행 유도) |
|---|---|---|
| 로딩 | KPI `—`, 표는 최소 스켈레톤 행(reduced-motion 시 정적 회색) | — (빠른 응답 전제, 스켈레톤은 짧게) |
| 신청 0건(전체) | 스푼체크 아웃라인 + 안내 | `아직 신청이 없어요. 랜딩에서 첫 신청이 들어오면 여기 나타납니다.` + 랜딩 링크 |
| 결과 0건(필터) | 빈 표 자리 안내 | `이 조건에 맞는 신청자가 없어요.` + `필터 지우기` |
| 조회 실패 | 인라인 배너 | `데이터를 불러오지 못했어요.` + `다시 시도` |
| 세션 만료 | 로그인으로 유도 | `세션이 만료됐어요. 다시 로그인해 주세요.` + `로그인` |

빈 화면은 분위기가 아니라 **다음 행동으로의 초대**다. 에러는 사과하지 않고, 무엇이 일어났고 어떻게 고치는지 인터페이스의 목소리로 말한다.

## 6. 시그니처 — "결정으로 내려가는 다리" (랭킹 → 명단 → 내보내기)

PRD의 표현대로 관리자는 **"데이터와 결정 사이의 다리"**다. 이 다리를 화면에서 **눈에 보이는 한 흐름**으로 만든 것이 이 페이지가 기억될 단 하나의 장면이다. 랜딩의 시그니처가 "결정의 순간"(접시 3D 플립)이었다면, 관리자의 시그니처는 **"결정을 실행으로 내려보내는 순간"**이다.

```
① 랭킹에서 '역삼동' 행 클릭
     └ 선택 행/1위 태그에 스푼체크 체크 드로잉 (--dur-reveal)   ← "이 동네로 정했다"
② 아래 목록이 '역삼동'으로 필터 (aria-live: "역삼동 312명")
     └ 목록 헤더에 활성 필터 칩 [선택됨: 역삼동 ✕], 페이지 목록으로 부드러운 스크롤
③ [필터 결과 CSV 내보내기] 버튼이 브랜드-소프트로 강조 (다음 행동 신호)
     └ 클릭 → 체크 드로잉 + "역삼동 312명을 내보냈어요"                ← 실행 완료
```

- **왜 이게 시그니처인가**: KR1(3클릭·10초 내 1위 동네)과 KR2(30초 내 CSV)의 경로가 *같은 화면에서 끊기지 않고* 이어진다. 탭 전환도, 화면 이동도 없다. 랜딩의 약속("먼저 엽니다") → 랭킹(무엇을) → 명단(누구에게) → CSV(어떻게)가 하나의 손동작으로 흐른다.
- **어휘의 연결**: 스푼체크 체크 드로잉은 랜딩·앱의 "결정 완료" 도장과 같은 마크(`SpoonMark draw`)다. 방문자가 "점심을 결정"하고, 앱이 "메뉴를 결정"하고, **운영자가 "출시를 결정"한다** — 같은 제스처, 커지는 무게.
- **절제(v1 범위 안)**: 이 다리는 새 데이터를 만들지 않는다. "발송 상태"나 "타깃 저장"은 컬럼이 필요하므로 v2(PRD §8). 체크 드로잉은 아무것도 저장하지 않는 **상태 전환 피드백**일 뿐 — 그래서 상시 모션 금지 원칙과 충돌하지 않는다. 프레이밍(언어·위계)과 클릭-필터 연결(기존 스키마)만으로 시그니처를 만든다.

## 7. 모션 시스템 요약 (최소)

| 트리거 | 효과 | 규칙 |
|---|---|---|

## 9. 접근성·품질 기준

- 표는 시맨틱 `<table>` + `<th scope>`. 정렬 헤더 `<button>` + `aria-sort`. 로딩·빈·에러 상태 명시적 텍스트.
- 파괴적 동작(숨기기)만 인페이지 모달(`role="dialog"` `aria-modal` 포커스 트랩 `Esc`). 지표·랭킹 변경은 조용히.
- 결과 수·필터 변경·토스트는 `aria-live="polite"`로 낭독(`"역삼동 312명"`).
- 포커스: `:focus-visible` 2px `--navy` 아웃라인(v2 동일).
- **색만으로 정보 전달 금지**: 랭킹 막대·중복·델타 모두 수치/라벨 병기. 부호(+/−)도 텍스트로.
- `prefers-reduced-motion: reduce` → 모든 transition/animation 제거, 스켈레톤 정적, 내용 즉시 반영.
- 색 대비: 잉크 상단바 위 흰 텍스트 AA 이상, 본문 `--ink`/`--surface` AA, `--sub`는 12px 이상에만.
- **보안이 곧 UX**(KR3): 비로그인 시 콘솔 UI 자체를 렌더하지 않고 로그인 게이트만. 데이터 호출은 세션 JWT + RLS로만 열림 — 클라이언트에 `service_role` 키 절대 없음.

## 10. 카피 원칙 & 핵심 문구

관리자의 목소리는 **차분하고 정확한 운영자**다 — 평이한 동사, 문장형 대소문자, 군더더기 없음. 각 요소는 정확히 한 가지 일만 한다.

- **원본 이름 금지**: `company` → **직장 동네**. `item`/`deleted_at` 같은 시스템 용어는 UI에 없다.
- **동작은 결과로 명명**: 버튼 `내보내기` → 토스트 `내보냈어요`. `숨기기` → `숨겼어요`. 같은 단어로 시작하고 끝낸다.
- **가역성을 언어에 담기**: 삭제가 아니라 "숨기기", 확인 문구에 "삭제되지 않고" "되돌릴 수 있어요"를 명시(PRD 확정 4의 안전 원칙을 카피로).
- **약속의 연결**: 랭킹 보조 문구 `신청 많은 순서대로 — 랜딩에서 약속한 그대로.` 로 이 화면이 왜 존재하는지 운영자에게 상기.
- **에러는 방향**: `데이터를 불러오지 못했어요. 다시 시도.` / `세션이 만료됐어요. 다시 로그인해 주세요.` — 사과·모호함 없이 무엇을·어떻게.

### 취한 위험 하나, 그 정당화

**슬림 잉크 상단바**를 상시 크롬으로 쓴 것 — 다크 헤더는 흔하지만, 여기선 임의의 스타일이 아니다. 랜딩에서 `--ink`는 "고민은 여기까지 — 결정"의 반전 밴드였고, 관리자는 그 결정이 *실제로* 내려지는 곳이다. 그 색을 크롬으로 입혀 두 화면을 의미로 잇되, 본문·표는 스캔이 쉬운 밝은 캔버스로 유지해 도구의 가독성을 지켰다(다크 테이블은 밀도 높은 데이터에 불리하므로 반전은 크롬에만). 대담함은 랭킹 시그니처 한 곳에 쓰고, 상단바는 조용히 — 나머지는 절제.
