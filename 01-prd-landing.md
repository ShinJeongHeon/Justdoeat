# PRD — Just do eat 랜딩페이지

## 1. 요약 (Summary)

점심 메뉴 자동 추천 서비스 **Just do eat**을 소개하는 랜딩페이지를 만든다.
핵심 전략은 "설명하지 말고, 겪게 하자" — 기능 나열 대신 히어로에 **실제로 움직이는 미니 데모**를 심어,
방문자가 3초 안에 "고민이 사라지는 경험"을 체감하고 앱(`index.html`)으로 넘어가게 한다.

## 2. 이해관계자 (Contacts)

| 이름 | 역할 | 비고 |
|---|---|---|
| 제품 오너 (사용자) | 기획·최종 승인 | 디자인 방향 결정권자. AI 디폴트 룩·포스터풍 거부, 정돈된 제품형 UI 선호 |
| Claude Code | 설계·구현 | 본 PRD 작성 및 랜딩페이지 개발 담당 |

## 3. 배경 (Background)

- **제품 현황**: Just do eat은 위치·예산·알레르기·일정·날씨·과거 피드백을 종합해
  매일 점심 TOP 3를 이유와 함께 추천하는 웹 앱. 단일 `index.html`로 설치 없이 즉시 동작하는
  프로토타입이 완성되어 있고, 디자인 시스템 v2(스푼체크 로고, 히어로+조건 패널 레이아웃)가 확정됨.
- **왜 지금인가**: 제품은 완성됐지만 "이게 뭔지, 왜 좋은지"를 처음 보는 사람에게 전달할 창구가 없다.
  앱 화면은 이미 조건이 설정된 사용자를 위한 것이라, 첫 방문자용 소개·설득 페이지가 별도로 필요하다.
- **가능해진 것**: 앱이 의존성 없는 단일 HTML이므로, 랜딩페이지도 같은 방식으로 만들면
  추천 엔진의 축소판을 랜딩 안에 직접 심을 수 있다. 스크린샷이 아닌 **살아있는 데모**가 가능하다.

## 4. 목표 (Objective)

**목표**: 처음 온 방문자가 서비스의 가치("점심 고민 제거")를 스스로 체감하고, 앱을 실행해 보게 만든다.

**왜 중요한가**: 이 서비스의 가치는 "선택 제거"다. 랜딩페이지가 길고 복잡하면
그 자체가 가치와 모순된다. 짧고, 만져볼 수 있고, 결론이 명확한 페이지여야
제품 철학("Just do eat")을 페이지 자체로 증명할 수 있다.

**핵심 결과 (Key Results)** — 프로토타입 단계이므로 v1은 정성 검증, 수치는 분석 도구 연동 후 측정:

| KR | 목표 | 측정 방법 |
|---|---|---|
| KR1 | 첫 화면만 보고 "무슨 서비스인지" 답할 수 있다 | 지인 5명 대상 5초 테스트 (v1, 정성) |
| KR2 | 히어로 미니 데모 인터랙션 비율 ≥ 40% | 분석 도구 연동 후 (v2) |
| KR3 | CTA("지금 추천받기") 클릭률 ≥ 15% | 분석 도구 연동 후 (v2) |
| KR4 | 모바일 Lighthouse 성능 점수 ≥ 90 | 개발 완료 시점 즉시 측정 (v1) |

## 5. 시장 세그먼트 (Market Segments)

세그먼트는 인구통계가 아니라 **해결할 문제**로 정의한다.

- **주 세그먼트 — "결정 피로" 직장인**: 매일 11시 30분마다 "오늘 뭐 먹지"를 반복하는 사무실 근무자.
  문제 = 메뉴 선택에 쓰는 시간과 에너지. 랜딩 유입도 대부분 점심 전후 모바일로 예상 → **모바일 우선 설계**.
- **부 세그먼트 — 낯선 동네의 방문자**: 출장자·여행자. 문제 = 주변 정보 부족.
  위치 변경(출장 모드) 기능이 소구점이지만, 랜딩에서는 하단 한 줄로만 언급.
- **제약**:
  - 프로토타입이므로 서버·회원가입 없음. 랜딩의 약속이 앱의 실제 능력을 넘지 않아야 한다.
  - 앱과 같은 조건: 의존성 없는 정적 파일, 인터넷 없이도 페이지 자체는 동작.

## 6. 가치 제안 (Value Propositions)

랜딩에서 전달할 가치는 **3축으로 압축**한다. 그 외 기능은 나열하지 않는다.

| 축 | 고객의 일(Job) | 얻는 것 | 피하는 고통 | 경쟁 대비 우위 |
|---|---|---|---|---|
| ① 고민 제거 | "점심을 빨리 정하고 싶다" | 매일 TOP 3 + 이유 칩 | 무한 재추천의 선택 피로 | 무한 리스트가 아닌 "오늘의 결론" 제시 |
| ② 투명함 | "왜 이 추천을 믿어야 하지?" | 점수 구성·경쟁 식당 비교 전체 공개 | 근거 없는 추천에 대한 불신 | **"왜 이 집이에요?" 패널은 타 추천 앱에 없음** |
| ③ 나를 학습 | "내 취향을 기억해 줬으면" | 식사 달력·피드백 학습·월간 리포트 | 매번 같은 조건을 다시 입력 | 먹을수록 정확해지는 개인화 |

신뢰 보강 장치: **당일 인증 리뷰** — 그날 결정한 사람만, 그날만 리뷰 작성 가능.
허위 리뷰가 구조적으로 불가능하다는 점을 별도 섹션으로 짧고 강하게 전달.

## 7. 솔루션 (Solution)

### 7.1 UX / 페이지 구조 (위 → 아래)

```
[1] 히어로
    스푼체크 로고 + "오늘 점심, 바로 결정"
    인터랙티브 미니 데모: 조건 칩(날씨/예산/매운맛 등)을 누르면
    추천 카드 TOP 3가 실시간으로 바뀜
    대형 CTA "지금 추천받기" → index.html

[2] 공감 한 줄 (한 문단)
    "인생은 선택의 연속. 점심만큼은 고민하지 마세요."

[3] 작동 방식 3단계
    ① 내 조건 (예산·알레르기·일정·날씨 자동 감지)
    ② 매일 TOP 3 + 이유 칩
    ③ 결정 → 기록 → 피드백으로 점점 똑똑해짐

[4] 킬러 피처 딥다이브 — "왜 이 집이에요?"
    실제 점수 패널 UI 재현 (가감점 구성 + 경쟁 식당 비교)

[5] 신뢰 장치 — 당일 인증 리뷰
    "그날 결정한 사람만, 그날만" 구조 설명 (짧게)

[6] 기록·리포트 미리보기
    식사 달력 + 월간 리포트 카드 목업

[7] 나머지 기능 아이콘 그리드 (한 줄)
    혼잡도 · 영양정보 · 원산지 · 출장 모드 · 지도 등

[8] 마무리 CTA
    "Just do eat" 반복 + CTA 재노출
```

### 7.2 핵심 기능 (랜딩페이지 자체의 기능)

1. **히어로 미니 데모**: 추천 엔진의 단순화 버전을 인라인 JS로 구현.
   조건 칩 토글 → 예시 식당 풀에서 점수 재계산 → 카드 순위·이유 칩 갱신.
   실제 앱 로직의 축소판이되, 랜딩용 고정 데이터 6~8곳으로 충분.
2. **v2 디자인 시스템 그대로 확장**: 랜딩과 제품의 얼굴이 같아야 신뢰가 생긴다.
   - 팔레트: 배경 `#F2F3EF`, 카드 `#FFF`, 잉크 네이비 `#1D2A38`, 고추장 레드 `#E0431F`/`#B5300F`
   - 타입: 디스플레이·CTA = Black Han Sans, 본문 = Pretendard(tabular-nums), `word-break: keep-all`
   - 스푼체크 SVG 심볼(#spoonCheck) 재사용, 장식 이모지 금지, radius 10~16px, 은은한 그림자
3. **모바일 우선 반응형**: 모바일 단일 컬럼 기준으로 설계 후 데스크톱 확장.
4. **접근성·품질**: 토글은 `role="button"` + 키보드 지원, 헤드리스 Chrome 스크린샷으로 검증.

### 7.3 기술

- **의존성 없는 단일 `landing.html`** (앱과 동일한 방식, 파일은 분리).
- `index.html`의 CSS 토큰·로고 심볼을 복사해 룩 일관성 자동 확보.
- 외부 요청 없음(폰트 제외) → 오프라인에서도 페이지 동작, Lighthouse 성능 확보 용이.

### 7.4 가정 (검증 필요)

- [ ] 미니 데모가 스크린샷보다 전환에 효과적이다 → 5초 테스트·데모 인터랙션율로 검증
- [ ] 3축 압축으로도 서비스 가치가 충분히 전달된다 (기능 12개 나열 불필요) → 5초 테스트
- [ ] 방문자 다수가 모바일이다 → 분석 도구 연동 후 확인
- [ ] 랜딩 → 앱 이동 시 맥락 손실이 없다 (별도 온보딩 불필요) → 이동 직후 이탈 관찰

## 8. 릴리스 (Release)

**v1 (첫 버전 — 짧은 개발 사이클 1회)**
- 섹션 [1]~[8] 전체, 히어로 미니 데모(칩 토글 → 카드 갱신 수준), 모바일/데스크톱 반응형
- 완료 기준: 5초 테스트 통과, Lighthouse 성능 ≥ 90, 헤드리스 스크린샷 검증

**v2 (후속)**
- 간단한 분석(CTA 클릭·데모 인터랙션 카운트) 연동 → KR2·KR3 측정 시작
- 미니 데모 고도화(실제 엔진 로직 공유), OG 태그·공유 카드 등 배포 준비

**범위 제외 (하지 않는 것)**
- 기능 전체 나열식 스펙시트 섹션
- 앱과 다른 별도 룩(포스터풍·크림+테라코타·이모지 장식)
- 회원가입·서버 연동을 전제로 한 약속 (프로토타입 능력 밖의 과장 금지)
